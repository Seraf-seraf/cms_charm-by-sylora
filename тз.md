# Техническое задание на разработку интернет-магазина ручных украшений на OpenCart

## 1. Общая информация

### 1.1. Название проекта

Интернет-магазин ручных украшений.

Рабочее название бренда: `Charm by Sylora`.

### 1.2. Назначение сайта

Создать современный, легкий и удобный интернет-магазин на CMS OpenCart для продажи ручных украшений: серег, браслетов, колье, подвесок, комплектов и других авторских изделий.

Сайт должен:

- продавать товары через каталог и корзину;
- принимать онлайн-оплату через отдельный payment-service и подключенного платежного провайдера;
- корректно работать на мобильных устройствах, планшетах и десктопах;
- иметь светлую и темную тему;
- быть подготовленным к SEO-продвижению и индексации поисковыми системами;
- быть масштабируемым для дальнейшего добавления категорий, фильтров, акций, интеграций, способов доставки и контентных страниц.

### 1.3. Язык сайта

Основной и единственный язык сайта: русский.

Мультиязычность не требуется. Переключатели языков, валют и лишние языковые URL не выводить, если заказчик отдельно не запросит их позднее.

### 1.4. CMS

Сайт разрабатывается на CMS OpenCart.

Версию OpenCart выбрать перед началом разработки с учетом совместимости:

- актуальной стабильной версии OpenCart;
- payment-service и платежного провайдера;
- модулей доставки;
- PHP-версии на хостинге;
- доступных SEO-модулей;
- требований безопасности.

Если совместимость популярных модулей лучше у OpenCart 3.x, допускается использовать OpenCart 3.x. Если все необходимые модули стабильно поддерживают OpenCart 4.x, предпочтительно использовать OpenCart 4.x. Итоговую версию зафиксировать в техническом отчете.

## 2. Цели проекта

### 2.1. Бизнес-цели

- Запустить интернет-магазин ручных украшений с возможностью оформления и оплаты заказа.
- Сформировать визуальный образ бренда: современный, теплый, авторский, аккуратный, без визуальной перегруженности.
- Сделать каталог, в котором удобно выбирать украшения по типу, цвету, материалу, цене и наличию.
- Подготовить сайт к органическому продвижению в Яндекс и Google.
- Обеспечить возможность дальнейшего роста ассортимента и функциональности без полной переделки сайта.

### 2.2. Пользовательские цели

Посетитель должен быстро:

- понять, что продает магазин;
- посмотреть новинки и популярные украшения;
- перейти в каталог;
- отфильтровать товары;
- открыть карточку товара;
- увидеть фото, цену, описание, материалы, наличие и условия доставки;
- добавить товар в корзину;
- оформить заказ;
- оплатить заказ;
- найти информацию об авторе, контактах, доставке, оплате и возврате.

## 3. Целевая аудитория

Основная аудитория:

- женщины 18-45 лет;
- покупатели подарков;
- люди, интересующиеся ручной работой, авторскими украшениями, минималистичными и индивидуальными изделиями;
- пользователи мобильных устройств.

Требования к восприятию:

- сайт должен выглядеть доверительно и аккуратно;
- оформление должно подчеркивать ручную работу и индивидуальность изделий;
- интерфейс должен быть простым, без перегрузки декоративными элементами;
- покупка должна занимать минимальное количество шагов.

## 4. Основные ограничения и принципы разработки

### 4.1. Ядро OpenCart

Запрещено изменять ядро CMS напрямую.

Все доработки выполнять через:

- дочернюю или отдельную тему;
- шаблоны темы;
- модули;
- события OpenCart;
- OCMOD/vQmod только при необходимости и с документированием;
- отдельные JS/CSS-файлы;
- настройки админ-панели.

Любые изменения, которые затрагивают системные файлы, должны быть обоснованы и описаны в отчете.

### 4.2. Масштабируемость

Архитектура должна позволять:

- добавлять новые категории товаров;
- расширять фильтры каталога;
- подключать новые способы оплаты;
- подключать новые способы доставки;
- добавлять промокоды и акции;
- добавлять SEO-посадочные страницы;
- добавлять контентные страницы без изменения общей верстки;
- менять цветовую схему через CSS-переменные;
- обновлять CMS и модули с минимальным риском для темы;
- использовать отдельный payment-service как независимый слой оплаты, чтобы OpenCart был одним из клиентов сервиса, а не местом хранения всей платежной логики.

### 4.3. Приоритет технологий

Желательно использовать:

- HTML/Twig-шаблоны OpenCart;
- CSS;
- CSS Custom Properties;
- JavaScript;
- минимальное количество внешних frontend-зависимостей.

Допускается:

- SCSS/Sass, если в проекте будет настроена сборка;
- Vanilla JS;
- jQuery только там, где это обусловлено OpenCart или установленными модулями;
- сборка через Vite/Webpack/Gulp, если это оправдано и задокументировано.

Не использовать тяжелые frontend-фреймворки для базовой витрины, если они не нужны для конкретной функции.

## 5. Цветовая палитра и визуальный стиль

### 5.1. Основная палитра

Использовать базовые цвета:

- розовый: `#d192b2`;
- коричневый: `#785b44`;
- зеленый: `#6da16e`.

Цвета должны применяться не как равномерная заливка всего интерфейса, а как акценты.

Рекомендуемая роль цветов:

- `#d192b2` - основной брендовый акцент, кнопки, активные элементы, декоративные линии;
- `#785b44` - текстовые и теплые нейтральные акценты, заголовки, элементы доверия;
- `#6da16e` - статусы, наличие, подтверждение, экологичность, вторичные CTA.

### 5.2. Светлая тема

Светлая тема должна быть основной.

Пример семантических CSS-переменных:

```css
:root {
  --color-brand: #d192b2;
  --color-earth: #785b44;
  --color-leaf: #6da16e;
  --color-bg: #ffffff;
  --color-surface: #f8f5f3;
  --color-surface-2: #fff9fb;
  --color-text: #2a2421;
  --color-muted: #6f625c;
  --color-border: #e7ddd8;
  --color-success: #6da16e;
  --color-danger: #b84a5a;
}
```

### 5.3. Темная тема

Темная тема должна быть полноценной, а не простым инвертированием цветов.

Пример семантических CSS-переменных:

```css
[data-theme="dark"] {
  --color-brand: #d192b2;
  --color-earth: #c7aa94;
  --color-leaf: #83bd84;
  --color-bg: #171312;
  --color-surface: #211b19;
  --color-surface-2: #2a211f;
  --color-text: #f5efea;
  --color-muted: #c8b9b0;
  --color-border: #3a302c;
  --color-success: #83bd84;
  --color-danger: #e17888;
}
```

### 5.4. Переключение темы

Реализовать переключатель светлой/темной темы:

- в шапке сайта;
- доступный с клавиатуры;
- с понятной иконкой;
- с сохранением выбора пользователя в `localStorage`;
- с учетом системной темы пользователя через `prefers-color-scheme`, если пользователь не сделал явный выбор.

Логика:

1. Если в `localStorage` есть выбор, использовать его.
2. Если выбора нет, использовать системную тему.
3. Если системная тема недоступна, использовать светлую тему.

### 5.5. Общий визуальный стиль

Стиль: современный, чистый, теплый, ремесленный, премиальный без перегруза.

Необходимо:

- много свободного пространства;
- аккуратная сетка;
- крупные качественные фото товаров;
- мягкие тени только там, где они помогают отделить элементы;
- радиусы скругления умеренные, ориентир 6-8 px;
- без визуального шума;
- без большого количества декоративных плашек;
- без агрессивных анимаций;
- без перегруженных градиентов;
- без кислотных цветов.

## 6. Структура меню

В основном меню должны быть пункты:

1. Главная
2. Каталог
3. Обо мне
4. Контакты
5. Корзина

### 6.1. Поведение меню

На десктопе:

- горизонтальное меню в шапке;
- активный пункт выделяется цветом или нижней линией;
- корзина может быть отдельной иконкой с количеством товаров.

На мобильных:

- меню открывается через кнопку;
- пункты крупные, удобные для нажатия;
- корзина доступна из шапки без необходимости открывать меню;
- закрытие меню по клику вне меню, по Escape и по выбору пункта.

### 6.2. Шапка сайта

Шапка должна содержать:

- логотип или текстовый знак бренда;
- основное меню;
- кнопку поиска или поле поиска;
- переключатель темы;
- ссылку/иконку корзины с количеством товаров;
- мобильную кнопку меню.

Шапка должна быть компактной и не занимать чрезмерно много места.

Допустимо сделать шапку липкой при прокрутке, если это не мешает просмотру товаров на мобильных устройствах.

## 7. Раздел "Главная"

Главная страница должна сразу показывать тематику магазина и направлять пользователя к покупке.

### 7.1. Первый экран

Содержимое:

- название бренда;
- короткий оффер: ручные украшения, созданные с вниманием к деталям;
- главное фото или композиция из фото украшений;
- основная кнопка "Смотреть каталог";
- вторичная ссылка "О бренде" или "Обо мне".

Требования:

- первый экран не должен выглядеть как абстрактный лендинг без товаров;
- изображения должны показывать реальные украшения или качественные заглушки до загрузки финальных фото;
- на мобильных основной CTA должен быть виден без лишней прокрутки;
- текст не должен перекрывать важные части фото.

### 7.2. Блок категорий

Показать основные категории, например:

- Серьги;
- Браслеты;
- Колье;
- Подвески;
- Комплекты;
- Новинки;
- Подарки.

Категории должны выводиться карточками или компактной сеткой с фото.

### 7.3. Блок избранных товаров

Вывести товары:

- новинки;
- популярные;
- рекомендуемые;
- товары со скидкой, если есть.

Карточка товара должна содержать:

- фото;
- название;
- цену;
- наличие;
- кнопку "В корзину";
- ссылку на карточку товара.

### 7.4. Блок преимуществ

Короткий блок с 3-4 преимуществами:

- ручная работа;
- бережная упаковка;
- ограниченные коллекции;
- доставка по России;
- онлайн-оплата.

Не перегружать большим текстом.

### 7.5. Блок "О мастере"

Короткий фрагмент:

- фото автора или рабочего процесса;
- 2-3 предложения о бренде;
- ссылка на страницу "Обо мне".

### 7.6. SEO-текст на главной

Добавить небольшой SEO-текст внизу главной страницы.

Требования:

- текст должен быть естественным;
- без переспама ключевыми словами;
- объем ориентировочно 800-1500 знаков;
- должен описывать ручные украшения, материалы, стиль, подарочные сценарии и доставку.

## 8. Раздел "Каталог"

Каталог - ключевой раздел сайта.

### 8.1. Категории

Предусмотреть структуру категорий:

- Все украшения;
- Серьги;
- Браслеты;
- Колье;
- Подвески;
- Кольца, если ассортимент есть;
- Комплекты;
- Подарки;
- Новинки;
- Распродажа, если планируются скидки.

Категории должны быть управляемыми из админ-панели OpenCart.

### 8.2. Список товаров

На странице каталога должны быть:

- заголовок категории;
- краткое описание категории;
- фильтры;
- сортировка;
- переключение количества товаров на странице;
- сетка товаров;
- пагинация или подгрузка с учетом SEO.

Сетка:

- десктоп: 3-4 товара в ряд;
- планшет: 2-3 товара в ряд;
- мобильные: 1-2 товара в ряд в зависимости от ширины.

### 8.3. Фильтры

Минимальный набор фильтров:

- категория;
- цена;
- материал;
- цвет;
- тип украшения;
- наличие;
- новинки;
- скидки.

Если OpenCart-модуль фильтрации не покрывает требования, подобрать совместимый модуль или реализовать доработку без изменения ядра.

На мобильных фильтры должны открываться в боковой панели или модальном блоке.

### 8.4. Сортировка

Варианты сортировки:

- по популярности;
- по новизне;
- по цене по возрастанию;
- по цене по убыванию;
- по названию.

### 8.5. Карточка товара в каталоге

Карточка товара должна содержать:

- основное фото;
- второе фото при наведении на десктопе, если есть;
- название товара;
- цену;
- старую цену при скидке;
- бейдж "Новинка", "Скидка", "Нет в наличии" при необходимости;
- короткое указание наличия;
- кнопку "В корзину";
- кнопку быстрого просмотра или ссылку "Подробнее", если это не усложняет интерфейс.

Требования:

- карточки должны быть одинаковой высоты в одной строке;
- фото не должны сжиматься некрасиво;
- кнопки не должны прыгать при разной длине названия;
- на мобильных все элементы должны быть читаемыми.

### 8.6. Страница товара

Страница товара должна содержать:

- хлебные крошки;
- галерею фото;
- название товара;
- цену;
- наличие;
- артикул, если используется;
- материалы;
- размер;
- цвет;
- описание;
- рекомендации по уходу;
- условия доставки и возврата;
- выбор опций, если товар имеет варианты;
- количество;
- кнопку "В корзину";
- блок "Похожие товары" или "С этим смотрят";
- SEO-метаданные.

### 8.7. Фото товара

Требования:

- поддерживать несколько фото;
- адаптивная галерея;
- увеличение фото по клику;
- lazy loading для дополнительных изображений;
- alt-тексты для SEO.

### 8.8. Наличие товара

Для ручных украшений важно корректно отражать наличие.

Варианты:

- "В наличии";
- "Под заказ";
- "Нет в наличии";
- "Осталось мало".

Если товар создается вручную и может повторяться, использовать статус "Под заказ" с пояснением срока изготовления.

## 9. Раздел "Обо мне"

Страница должна рассказывать о мастере и бренде.

### 9.1. Содержимое

Разделы:

- краткая история бренда;
- фото мастера или рабочего процесса;
- философия: ручная работа, аккуратность, материалы, ограниченные коллекции;
- чем украшения отличаются;
- как создаются изделия;
- ссылка на каталог;
- блок доверия: упаковка, доставка, оплата, отзывы, если есть.

### 9.2. Дизайн

Страница должна быть спокойной и визуально легкой.

Не использовать длинную стену текста. Разбить контент на логические блоки с фото.

### 9.3. SEO

Для страницы настроить:

- уникальный Title;
- Description;
- H1;
- человекочитаемый URL, например `/about`;
- Open Graph для корректного отображения при отправке ссылки.

## 10. Раздел "Контакты"

### 10.1. Содержимое

Страница должна содержать:

- название магазина/бренда;
- email;
- телефон, если предоставлен;
- мессенджеры, если предоставлены;
- ссылки на социальные сети;
- город или регион работы;
- график обработки заказов;
- форму обратной связи;
- юридическую информацию, если магазин оформлен на ИП/ООО/самозанятого;
- блок "Доставка и оплата" или ссылки на отдельные страницы.

### 10.2. Форма обратной связи

Поля:

- имя;
- email или телефон;
- сообщение;
- согласие на обработку персональных данных.

Требования:

- валидация на клиенте и сервере;
- защита от спама;
- уведомление администратору на email;
- понятное сообщение после отправки;
- не раскрывать технические ошибки пользователю.

### 10.3. Карта

Если есть офлайн-точка или мастерская для самовывоза, добавить карту.

Если офлайн-точки нет, карту не добавлять, чтобы не создавать ложных ожиданий.

## 11. Раздел "Корзина"

### 11.1. Основные функции

Корзина должна позволять:

- просматривать добавленные товары;
- менять количество;
- удалять товар;
- видеть цену за единицу;
- видеть общую стоимость;
- видеть скидки, если применены;
- применять промокод, если включены промокоды;
- перейти к оформлению заказа;
- вернуться в каталог.

### 11.2. Мини-корзина

В шапке сайта должна быть мини-корзина:

- иконка корзины;
- количество товаров;
- краткая сумма;
- выпадающий список товаров или переход на страницу корзины.

После добавления товара:

- показать понятное уведомление;
- обновить счетчик без перезагрузки страницы, если технически возможно;
- предложить "Перейти в корзину" и "Продолжить покупки".

### 11.3. Оформление заказа

Оформление заказа должно быть простым.

Минимальные поля:

- имя;
- телефон;
- email;
- город;
- адрес доставки, если доставка до адреса;
- способ доставки;
- способ оплаты;
- комментарий к заказу;
- согласие с политикой конфиденциальности и условиями оферты.

Требования:

- понятные ошибки валидации;
- сохранение введенных данных при ошибке;
- адаптивная верстка;
- минимум лишних шагов;
- корректная работа гостевого заказа, если регистрация не обязательна.

Регистрацию пользователя не делать обязательной, если заказчик отдельно не потребует.

## 12. Оплата и payment-service

### 12.1. Общая схема оплаты

Оплата должна быть реализована через отдельный сервис оплаты `payment-service`.

OpenCart не должен напрямую интегрироваться с платежным провайдером как с основной точкой бизнес-логики. Для OpenCart необходимо разработать или подключить отдельное расширение оплаты, которое работает с `payment-service` через API.

Главное архитектурное правило:

```text
payment-service не знает про OpenCart как про CMS.
payment-service знает только merchant, payment, provider, webhook, notification.
OpenCart является merchant/client, который использует API payment-service.
```

Целевая схема:

```text
OpenCart payment extension
    -> POST /api/v1/payments
    -> payment-service
    -> payment provider
    <- payment_url

payment provider
    -> POST /api/v1/provider-webhooks/{provider_name}
    -> payment-service
    -> обновление статуса payment
    -> outbox/callback
    -> OpenCart callback endpoint
    -> смена статуса заказа
```

### 12.2. Выбор платежного провайдера

Платежный провайдер подключается внутри `payment-service`.

Если заказчик не указал провайдера до старта разработки, использовать рабочее допущение:

- основной вариант для РФ: ЮKassa;
- альтернативы: CloudPayments, Robokassa, PayKeeper, Тинькофф/Т-Банк эквайринг, СберБанк эквайринг.

Перед реализацией исполнитель должен проверить:

- поддержку тестового режима у провайдера;
- поддержку нужных способов оплаты;
- требования провайдера к webhook/callback и проверке подписи;
- совместимость с валютой и налоговыми настройками;
- подтверждение, что отдельная фискализация на стороне сайта не требуется, так как оплата работает через payment-service и платежного провайдера;
- необходимость передачи фискальных данных, параметров чека или онлайн-кассы, если выбранный провайдер потребует это отдельно.

### 12.3. Способы оплаты

Минимально предусмотреть:

- банковские карты;
- СБП, если поддерживается платежным провайдером;
- оплата при получении, если заказчик хочет оставить такой способ;
- ручной перевод только если заказчик явно разрешит.

### 12.4. Требования к payment-service

Требования:

- сервис должен быть отдельным Go-приложением, не завязанным на OpenCart;
- сервис должен хранить клиентов/мерчантов в БД;
- сервис должен создавать платежи через подключенных платежных провайдеров;
- сервис должен принимать webhook от провайдеров;
- сервис должен идемпотентно обновлять статусы платежей;
- сервис должен уведомлять клиента через `callback_url`;
- PostgreSQL должен быть источником истины по платежам и событиям;
- создание платежа должно быть идемпотентным по `Idempotency-Key` или вычисленному ключу заказа;
- повторный webhook от провайдера не должен создавать повторных действий;
- доставка callback в OpenCart должна выполняться через outbox/retry-механизм, чтобы временная недоступность магазина не теряла факт оплаты;
- сервис должен иметь health endpoint;
- миграции БД сервиса должны запускаться отдельно от приложения;
- секреты, API keys, shared secrets, provider secret keys и authorization headers нельзя логировать или хранить в открытом виде без необходимости.

Минимальные сущности payment-service:

- merchant;
- payment;
- provider;
- webhook event;
- outbox event.

Минимальные API payment-service:

- `POST /api/v1/payments`;
- `GET /api/v1/payments/{payment_id}`;
- `POST /api/v1/provider-webhooks/{provider_name}`;
- `GET /api/v1/health`.

### 12.5. Требования к расширению OpenCart

Для OpenCart необходимо реализовать расширение оплаты, которое:

- добавляет способ оплаты через `payment-service` в checkout;
- хранит настройки merchant/API key/shared secret/base URL/success URL/fail URL/callback URL через админ-панель или конфигурацию;
- при подтверждении заказа создает заказ в OpenCart со статусом "Ожидает оплаты";
- отправляет подписанный запрос на `POST /api/v1/payments`;
- передает в payment-service номер заказа, сумму, валюту, описание и данные покупателя, необходимые для платежа;
- получает `payment_url` и перенаправляет пользователя на оплату;
- предоставляет callback endpoint для уведомлений от payment-service;
- проверяет подпись callback от payment-service;
- идемпотентно обрабатывает повторные callback;
- после успешной оплаты меняет статус заказа на "Оплачен" или иной согласованный статус;
- при отмене, ошибке или возврате меняет статус заказа на согласованный статус;
- пишет безопасные логи ошибок без вывода технических данных пользователю;
- не хранит данные банковских карт.

Расширение должно быть сделано без прямого изменения ядра OpenCart. Если OCMOD/vQmod или изменение системных файлов окажется необходимым, это должно быть отдельно обосновано и описано в отчете.

### 12.6. URL и сценарии оплаты

Необходимо настроить:

- success URL для возврата после успешной оплаты;
- fail URL для возврата после ошибки или отмены;
- provider webhook URL в payment-service;
- callback URL из payment-service в OpenCart;
- тестовый режим провайдера;
- статусы заказа в OpenCart;
- безопасные пользовательские сообщения на страницах успеха, ошибки и отмены.

Необходимо протестировать:

- успешную оплату;
- отмененную оплату;
- ошибку оплаты;
- повторный callback;
- отсутствие дубля заказа при повторной отправке уведомления;
- отсутствие дубля платежа при повторном запросе создания платежа;
- повторную доставку callback из outbox при временной недоступности OpenCart callback endpoint.

### 12.7. Статусы заказа

Минимальные статусы:

- Новый;
- Ожидает оплаты;
- Оплачен;
- В обработке;
- Собирается;
- Отправлен;
- Выполнен;
- Отменен;
- Возврат.

После успешной онлайн-оплаты заказ должен переходить в статус "Оплачен" или иной согласованный статус.

### 12.8. Фискализация

Отдельная фискализация на стороне сайта в базовом объеме не требуется.

В рамках проекта:

- оплата проходит через payment-service и выбранного платежного провайдера;
- OpenCart и payment-service не хранят банковские данные;
- сайт не подключает отдельную онлайн-кассу;
- OpenCart должен корректно передавать данные заказа в payment-service и получать статусы оплаты через callback;
- payment-service должен корректно передавать платежные данные провайдеру и получать статусы оплаты через webhook;
- отдельную передачу фискальных данных, настройку онлайн-кассы и параметров чека не выполнять.

Если в будущем заказчик выберет схему оплаты, при которой агрегатор потребует отдельную настройку чеков, онлайн-кассы или фискальных параметров, это считается отдельным согласованием вне текущего базового объема.

## 13. Доставка

Хотя доставка не указана как отдельный пункт меню, интернет-магазин должен иметь рабочие способы доставки.

### 13.1. Минимальные способы

Предусмотреть:

- самовывоз, если доступен;
- доставка по России;
- курьерская доставка, если есть;
- СДЭК/Boxberry/Почта России при наличии выбранного модуля;
- ручной расчет доставки как временный вариант, если точные тарифы еще не готовы.

### 13.2. Требования

- стоимость доставки должна отображаться до оплаты;
- пользователь должен понимать сроки и условия;
- способ доставки должен сохраняться в заказе;
- администратор должен видеть выбранный способ в админ-панели;
- если доставка зависит от города, веса или суммы заказа, логика должна быть настроена и протестирована.

## 14. SEO: техническая настройка, продвижение и индексация

SEO должно быть выполнено как обязательная часть проекта, а не как опциональная доработка.

### 14.1. Техническое SEO

Настроить:

- ЧПУ URL в OpenCart;
- корректный `.htaccess` или серверные правила для SEO URL;
- единый основной домен;
- редирект с `http` на `https`;
- редирект с дубля `www` или без `www` на основной вариант;
- SSL-сертификат;
- канонические URL;
- отсутствие дублей главной страницы;
- отсутствие индексации служебных страниц;
- корректные коды ответа 200/301/404;
- страницу 404 в стиле сайта;
- файл `robots.txt`;
- файл `sitemap.xml`;
- автоматическое обновление sitemap при изменении товаров и категорий;
- микроразметку schema.org;
- Open Graph;
- фавиконки;
- корректные alt у изображений.

### 14.2. Индексация

После запуска:

- добавить сайт в Яндекс.Вебмастер;
- добавить сайт в Google Search Console;
- подтвердить права на сайт;
- отправить `sitemap.xml`;
- проверить файл `robots.txt`;
- запросить переобход главной страницы;
- запросить переобход ключевых категорий;
- проверить, что страницы товаров доступны для индексации;
- проверить отчет по ошибкам индексирования;
- исправить найденные ошибки.

### 14.3. robots.txt

Файл должен:

- разрешать индексацию главной, категорий, товаров, информационных страниц;
- закрывать корзину, оформление заказа, аккаунт, поиск с параметрами, системные страницы;
- указывать путь к sitemap.

Примерная логика:

```txt
User-agent: *
Disallow: /admin/
Disallow: /catalog/
Disallow: /system/
Disallow: /index.php?route=checkout/
Disallow: /index.php?route=account/
Disallow: /index.php?route=product/search
Disallow: /*?sort=
Disallow: /*?order=
Disallow: /*?limit=
Disallow: /*?page=
Allow: /
Sitemap: https://example.ru/sitemap.xml
```

Перед публикацией проверить правила под фактическую структуру OpenCart и модулей.

### 14.4. sitemap.xml

В sitemap включить:

- главную страницу;
- категории;
- товары;
- страницу "Обо мне";
- страницу "Контакты";
- страницы доставки, оплаты, возврата, политики конфиденциальности, если они созданы.

Не включать:

- корзину;
- оформление заказа;
- личный кабинет;
- служебные страницы;
- страницы с параметрами сортировки и фильтрации, если для них не создана отдельная SEO-логика.

### 14.5. Метатеги

Для каждой индексируемой страницы настроить:

- Title;
- Description;
- H1;
- человекочитаемый URL;
- canonical;
- Open Graph title;
- Open Graph description;
- Open Graph image.

Правила:

- Title уникальный, ориентировочно до 60-70 символов;
- Description уникальный, ориентировочно до 140-170 символов;
- H1 один на страницу;
- не дублировать Title и H1 полностью на всех страницах;
- не использовать спамные перечисления ключей.

### 14.6. Примеры SEO-шаблонов

Главная:

- Title: `Ручные украшения [Название бренда] - авторские украшения с доставкой`
- Description: `Авторские украшения ручной работы: серьги, браслеты, подвески и комплекты. Бережная упаковка, онлайн-оплата и доставка по России.`
- H1: `Ручные украшения [Название бренда]`

Категория:

- Title: `[Категория] ручной работы - купить в [Название бренда]`
- Description: `Выберите [категория] ручной работы: авторские изделия, качественные материалы, аккуратная упаковка и доставка по России.`
- H1: `[Категория] ручной работы`

Товар:

- Title: `[Название товара] - купить авторское украшение ручной работы`
- Description: `[Название товара]: материалы, размер, особенности. Онлайн-оплата, аккуратная упаковка и доставка по России.`
- H1: `[Название товара]`

### 14.7. Микроразметка

Добавить schema.org:

- `Organization` или `LocalBusiness` для магазина;
- `WebSite` с поиском, если поиск доступен;
- `BreadcrumbList`;
- `Product`;
- `Offer`;
- `AggregateRating` только если есть реальные отзывы;
- `Review` только для реальных отзывов.

Для товара разметить:

- название;
- изображение;
- описание;
- SKU, если есть;
- бренд;
- цену;
- валюту;
- наличие;
- URL.

### 14.8. Контентное SEO

Создать или подготовить структуру контентных страниц:

- Доставка и оплата;
- Возврат и обмен;
- Уход за украшениями;
- Размеры и материалы;
- Подарочная упаковка;
- Политика конфиденциальности;
- Пользовательское соглашение или оферта.

Если заказчик не готов публиковать все страницы сразу, заложить их в структуру и закрыть пустые страницы от индексации до наполнения.

### 14.9. SEO-тексты для категорий

Для основных категорий подготовить короткие SEO-описания:

- 500-1200 знаков;
- естественный русский язык;
- описание типа украшения, материалов, стиля, сценариев использования;
- без переспама;
- с внутренними ссылками на смежные категории, если уместно.

### 14.10. Продвижение после запуска

Базовый план продвижения:

- проверить индексацию через 7-14 дней после запуска;
- исправить ошибки из Яндекс.Вебмастера и Google Search Console;
- добавить уникальные описания для всех категорий;
- добавить уникальные описания для ключевых товаров;
- заполнить alt у изображений;
- создать страницы доверия: доставка, оплата, возврат, уход;
- настроить Яндекс.Метрику;
- настроить цели: добавление в корзину, начало оформления, успешный заказ, отправка формы;
- при необходимости настроить Google Analytics 4;
- подключить пиксели рекламных систем только по отдельному решению заказчика;
- добавить бренд в профили социальных сетей и указать ссылку на сайт;
- не использовать массовую закупку ссылок и другие рискованные методы.

SEO-продвижение не должно обещать гарантированные позиции. Результат зависит от конкуренции, контента, ассортимента, ссылочного профиля, поведения пользователей и регулярности работ.

## 16. Производительность

### 16.1. Требования

Сайт должен быть быстрым на мобильных устройствах.

Ориентиры:

- главная страница загружается без тяжелых блокирующих скриптов;
- изображения оптимизированы;
- lazy loading включен для изображений ниже первого экрана;
- CSS и JS минифицированы в production;
- критичные стили загружаются без заметной задержки;
- неиспользуемые модули OpenCart отключены;
- кеширование включено;
- шрифты подключены оптимально.

### 16.2. Изображения

Требования:

- использовать WebP/AVIF, если поддерживается инфраструктурой;
- сохранять fallback для JPG/PNG при необходимости;
- генерировать несколько размеров изображений;
- не загружать полноразмерные фото в карточки каталога;
- задавать `width` и `height` или стабильное соотношение сторон;
- использовать alt-тексты.

### 16.3. Проверка

После разработки проверить:

- PageSpeed Insights;
- Lighthouse;
- мобильную скорость;
- отсутствие layout shift;
- размер главной страницы;
- размер страниц каталога и товара.

## 19. Динамика и интерактивность

### 19.1. Общие требования

Интерактивность должна помогать покупке, а не отвлекать.

Реализовать:

- плавное открытие мобильного меню;
- плавное открытие фильтров на мобильных;
- обновление мини-корзины после добавления товара;
- уведомление о добавлении товара в корзину;
- переключение темы без перезагрузки;
- галерею товара;
- раскрывающиеся блоки на карточке товара: описание, уход, доставка;
- валидацию форм;
- маску телефона, если это не ухудшает ввод;
- состояние загрузки для кнопок при отправке форм и оформлении заказа.

### 19.2. Анимации

Анимации:

- короткие;
- не более 150-250 ms для интерфейсных элементов;
- отключаемые через `prefers-reduced-motion`;
- без чрезмерных параллаксов и тяжелых эффектов.

## 20. Страницы доверия и юридические страницы

Создать или предусмотреть:

- Доставка и оплата;
- Возврат и обмен;
- Политика конфиденциальности;
- Согласие на обработку персональных данных;
- Оферта или пользовательское соглашение;
- Контакты и реквизиты, если применимо.

Тексты юридических страниц должен предоставить заказчик или юрист. Исполнитель может подготовить технические шаблоны, но не должен выдавать их за юридически проверенные документы.

## 21. Админ-панель и управление контентом

Администратор должен иметь возможность:

- добавлять товары;
- редактировать товары;
- загружать фото;
- менять цены;
- менять наличие;
- добавлять категории;
- редактировать SEO-метаданные;
- редактировать информационные страницы;
- просматривать заказы;
- менять статусы заказов;
- видеть данные оплаты и доставки;
- управлять промокодами;
- управлять баннерами/главной страницей без обращения к разработчику, если это возможно через модули.

### 21.1. Структура товара

Для товара предусмотреть поля:

- название;
- категория;
- цена;
- старая цена;
- наличие;
- статус "под заказ";
- артикул;
- материалы;
- цвет;
- размер;
- вес, если влияет на доставку;
- описание;
- рекомендации по уходу;
- фото;
- SEO Title;
- SEO Description;
- SEO URL;
- alt для изображений.

## 22. Безопасность

Требования:

- использовать HTTPS;
- изменить стандартный путь админ-панели, если возможно и уместно;
- использовать сложные пароли;
- ограничить права пользователей админ-панели;
- обновить OpenCart и модули до безопасных версий;
- не хранить секреты в репозитории;
- не выводить технические ошибки пользователю;
- настроить резервное копирование;
- защитить формы от спама;
- проверить права на файлы и директории;
- отключить ненужные модули;
- проверить загрузку файлов через админ-панель;
- настроить регулярные бэкапы базы данных и файлов.

## 23. Email и уведомления

Настроить:

- SMTP-отправку писем;
- письмо пользователю после заказа;
- письмо администратору после заказа;
- письмо при смене статуса заказа;
- письмо после отправки формы обратной связи.

Письма должны:

- быть на русском языке;
- содержать номер заказа;
- содержать состав заказа;
- содержать сумму;
- содержать способ доставки и оплаты;
- не содержать лишней технической информации.

Опционально:

- уведомления в Telegram для администратора;
- уведомления о низком остатке товара.

## 24. Поиск

На сайте должен быть поиск по товарам.

Требования:

- поиск доступен из шапки;
- поиск работает по названию товара;
- желательно учитывать описание, артикул, материалы;
- страница результатов поиска адаптивная;
- пустой результат содержит понятное сообщение и ссылку в каталог;
- страницы поиска с параметрами не должны создавать SEO-дубли.

## 25. Избранное и сравнение

OpenCart часто содержит функции избранного и сравнения.

Для данного магазина:

- избранное можно оставить, если оно аккуратно встроено в дизайн;
- сравнение товаров можно скрыть, если оно не нужно для украшений и перегружает интерфейс;
- любые неиспользуемые элементы OpenCart убрать из витрины.

## 26. Отзывы

Отзывы можно включить, если заказчик готов модерировать их.

Если отзывы включены:

- отображать только одобренные отзывы;
- добавить защиту от спама;
- не добавлять фейковые отзывы;
- использовать микроразметку отзывов только для реальных отзывов.

Если отзывов пока нет:

- не показывать пустой блок как ключевой элемент;
- не выводить звездный рейтинг без данных.

## 27. Акции и промокоды

Предусмотреть:

- скидочную цену товара;
- бейдж скидки;
- промокоды через стандартный механизм OpenCart или модуль;
- отображение скидки в корзине;
- корректный расчет итоговой суммы перед оплатой.

## 32. Тестовые сценарии приемки

### 32.1. Навигация

- Пользователь открывает главную страницу.
- Видит бренд, CTA и товары/категории.
- Переходит в каталог.
- Переходит в "Обо мне".
- Переходит в "Контакты".
- Переходит в корзину.
- На мобильном открывает и закрывает меню.

### 32.2. Тема

- Пользователь включает темную тему.
- Обновляет страницу.
- Темная тема сохраняется.
- Пользователь включает светлую тему.
- Обновляет страницу.
- Светлая тема сохраняется.
- При отсутствии сохраненного выбора применяется системная тема.

### 32.3. Каталог

- Пользователь открывает категорию.
- Применяет фильтр по цене.
- Применяет фильтр по материалу.
- Меняет сортировку.
- Переходит на карточку товара.
- Возвращается назад без потери критичной навигации.

### 32.4. Товар

- Пользователь смотрит фото.
- Открывает увеличенное изображение.
- Выбирает опцию, если она есть.
- Добавляет товар в корзину.
- Видит уведомление.
- Счетчик корзины обновляется.

### 32.5. Корзина

- Пользователь меняет количество.
- Удаляет товар.
- Применяет промокод, если промокоды включены.
- Видит корректную итоговую сумму.
- Переходит к оформлению.

### 32.6. Оформление заказа

- Пользователь заполняет обязательные поля.
- Выбирает доставку.
- Выбирает оплату.
- Подтверждает согласие с условиями.
- Создает заказ.
- Получает письмо.
- Администратор получает письмо.

### 32.7. Оплата

- Расширение OpenCart создает платеж через payment-service.
- Пользователь перенаправляется на `payment_url`.
- Успешная оплата меняет статус заказа.
- Неуспешная оплата не помечает заказ как оплаченный.
- Повторный callback не создает дубль оплаты.
- Повторное создание платежа по тому же заказу не создает дубль платежа.
- Webhook провайдера обновляет статус платежа в payment-service.
- Callback от payment-service обновляет статус заказа в OpenCart.
- Временная недоступность callback endpoint в OpenCart приводит к повторной доставке уведомления.
- Отмена оплаты возвращает пользователя на понятную страницу.

### 32.8. SEO

- Главная доступна по одному основному URL.
- `robots.txt` открывается.
- `sitemap.xml` открывается.
- Категории имеют ЧПУ URL.
- Товары имеют ЧПУ URL.
- У страниц есть Title и Description.
- У товара есть Product-разметка.
- У хлебных крошек есть BreadcrumbList.
- Закрытые страницы не попадают в sitemap.
